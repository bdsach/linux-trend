The OpenBSD project produces a free, multi-platform BSD 4.4-based UNIX-like operating system. Its efforts emphasize portability, standardisation, correctness, proactive security and integrated cryptography. The project also develops the widely-used and popular OpenSSH (OpenBSD Secure Shell) software, which provides encrypted communication sessions over a computer network using the SSH protocol.

 Let me start by saying that very often people comment that OpenBSD is not for beginners. That is only true if you are the kind of beginner who doesn't enjoy the challenge of learning something new and who gives up easily. If all you have ever done with a computer is switch on the power and hope that Windows or Android or iOS boots up so you can login, and you really aren't interested at all in how it works but rather just prefer to get things done, then yes, I agree that OpenBSD is not for you. If you are curious about how it really works, though, then time spent learning to install and set up a usable OpenBSD system is well worth it. I learned coming from Windows back in 2008 or so, and yes, it was initially difficult and sometimes frustrating, but I wouldn't trade it for just staying put on Windows and now feeling trapped 20 years later, as so many do.

OpenBSD was my daily driver for about five years until the mid-teens. At that time I moved to Linux because there was some functionality I needed that was difficult to impossible to get working on OpenBSD at the time. I eventually ended up using MX Linux, which is an excellent distro. I still have it on a USB stick just in case, and with the persistence tools, I can keep it up to date very easily. If you are a beginner to using an operating system that you install yourself and this seems too hard, I would highly recommend looking at MX Linux.

I decided to go back to OpenBSD for several reasons. I missed the simplicity of being able to configure pretty much everything using human-readable text files. There's some learning curve, sure, but it is not that difficult and the man pages are second to none. Also, there are a few good guides online to help if you're brand new to OpenBSD. (Calomel is to be avoided at all costs btw.)

The base is very clean and has useful programs to do every essential administrative task. Everything is built with security in mind. It is not an afterthought that involves hardening some processes and settings or even substituting a chunk of the codebase that was secured by the NSA. (SELinux makes me laugh every time I think about it. You're trusting the U.S. government agency whose sworn duty is to spy on all communications to secure your operating system? I suppose if I had the ability and time to audit the 2,000,000 lines of code in the kernel, I might think differently -- but I don't, so it is all based on trust.)

The kernel and userland are developed together on OpenBSD. This means that no one outside the dev team can decide to change the way all the various processes are initialized and controlled, and again, the settings are all available with text files, rather than trying to tweak the obscure internals of something as opaque as shoggothd.

Hardware support on OpenBSD has improved to the point where everything works great on my two-year-old laptop and there is a package for anything I need to do with a computer so the problems I was facing with using OpenBSD as my daily driver ten years ago have vanished.

Lastly, there is no cancerous copyleft code in base. It's all under the ISC / BSD / MIT / Apache licenses. The longer I do this, the more I realize how beneficial that is, if only to avoid the continuous preaching about how great the GPL and its variants are. There is an underlying ideological bent to everything done and said in Linuxland that becomes very tiresome if all you want to do is get something done with a computer. (I can already hear the reply that "not choosing a side is in itself an ideological position!" Please, stfu and go get a haircut instead.)

The base is very clean and has useful programs to do every essential administrative task. Everything is built with security in mind. It is not an afterthought that involves hardening some processes and settings or even substituting a chunk of the codebase that was secured by an agency of the U.S. government.

The kernel and userland are developed together on OpenBSD. This means that they are well-integrated. There are no surprises because some dependency broke upstream.

Hardware support on OpenBSD has improved to the point where everything works great on my two-year-old laptop and there is a package for anything I need to do with a computer so the problems I was facing with using OpenBSD as my daily driver ten years ago have vanished.

Lastly, there is no copyleft code in base. It's all under the ISC / BSD / MIT / Apache licenses. The longer I do this, the more I realize how beneficial that is.
